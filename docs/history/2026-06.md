## 2026-06-18 — Cloud Functions, gráficos, delete de cartão, UX da fatura

### Cloud Functions — automação server-side

Quatro funções scheduled no codebase `billing` (`southamerica-east1`), todas usando `getFirestore()` lazy (dentro do handler para não quebrar antes do `initializeApp()`):

- **`closeInvoicesDue`** (`0 0 * * *`): `collectionGroup('cards')` onde `closingDay == hoje && isActive == true`, fecha invoices `open` com `referenceMonth <= mêsAtual`, envia push com valor.
- **`generateRecurrences`** (`0 6 * * *`): `collectionGroup('recurring')` onde `isActive == true && nextOccurrenceAt <= agora`, cria transação + avança `nextOccurrenceAt` (weekly +7d, monthly +1m, yearly +1y). Pula regras sem `amountCents` ou `accountId`.
- **`sendDueReminders`** (`0 8 * * *`): `collectionGroup('bills')` onde `status == pending && dueDate` entre agora e +3 dias. Envia push com data de vencimento.
- **`sendDailyLogReminder`** (`0 20 * * *`): `collectionGroup('fcmTokens')` busca todos os tokens de todos os usuários, envia push em lotes de 500 com "Como foi o dia?" linkando para `/app/transactions/new`.

**Push** (`functions/src/push.ts`): `sendPushToUser(userId, title, body, link)` — busca `users/{uid}/fcmTokens`, envia `sendEachForMulticast`, remove tokens stale automaticamente.

**Client-side** (`src/pwa/notifications.ts`): pede permissão de notificação, obtém token FCM via VAPID key (`VITE_FIREBASE_VAPID_KEY`), salva em `users/{uid}/fcmTokens/{token}`. Registrado em `AppShell.tsx` quando `authFromCache` passa para `false`. Service worker gerado em build via plugin Vite.

**Deploy**: Stripe secrets configurados como placeholder no Secret Manager para permitir deploy sem ativar billing. `APP_BASE_URL` em `functions/.env.zerou-26757`.

### Fix: fatura aberta permanece open

**Problema**: `resolveInvoiceStatus` retornava `'paid'` ou `'partial'` mesmo para faturas com lifecycle `'open'`. Um pagamento total antecipado numa fatura ainda aberta a marcava prematuramente como "Paga" — errado, pois novas compras continuam entrando.

**Comportamento Nubank**: fatura em aberto = sempre "Aberta" até o fechamento, independente de pagamentos. Só após `closeInvoice` → lifecycle `'closed'` → aí resolve para `'partial'`/`'paid'`.

**Fix** em `resolveInvoiceStatus`:
```ts
if (input.lifecycle === 'open') return 'open'; // overpaid verificado antes
// lifecycle === 'closed' → resolve paid/partial/closed normalmente
```

**Cascatas**:
- `advance` no pagamento (`InvoicePage`): `invoice.status === 'open'` → sempre `true` enquanto aberta. Antes, após parcial ficava `false` (errado).
- `Comprometido`: filtrava por `outstandingBalanceCents > 0` — fatura totalmente coberta por antecipação (saldo = 0) some corretamente.
- Hint "Fatura em aberto — novos lançamentos entram aqui" aparece corretamente mesmo após pagamentos.

**Teste atualizado**: `calculateInvoice.test.ts` — cenário de `'partial'`/`'paid'` movido para `lifecycle: 'closed'`; novo caso confirma fatura aberta com advance payment permanecendo `'open'`. Total: 46 testes.

## 2026-06-17 — painel admin + Cloud Function de deleção

### Contexto
Pedido de central de admin completa acessível só para `a.thurcos@gmail.com`, com visão de todos os usuários, espaços de casal e convites, mais botão de deletar conta definitivamente.

### Decisões de arquitetura

**Dois codebases de Cloud Functions separados**

O codebase existente (`functions/`) usa `defineSecret` para as chaves do Stripe. Deploy via Firebase CLI em modo não-interativo exige que os segredos estejam pré-configurados no Secret Manager — o que bloqueia qualquer deploy enquanto o Stripe não estiver configurado. Solução: codebase separado `functions-admin/` sem secrets, deployável de forma independente.

`firebase.json` passou de um objeto simples para array de codebases:
```json
"functions": [
  { "source": "functions", "codebase": "billing", ... },
  { "source": "functions-admin", "codebase": "admin", ... }
]
```

Deploy seletivo: `npx firebase deploy --only functions:admin --project zerou-26757`

**Por que Cloud Function para deleção (não client-side)**

`auth.deleteUser(uid)` é exclusivo do Firebase Admin SDK — não existe no SDK do client. Sem uma CF, a deleção do Firebase Auth ficaria impossível. Toda a lógica de varredura do Firestore poderia ser feita client-side (o admin já tem `read` via regras), mas centralizar na CF é mais seguro e evita expansão das rules de `delete` para o admin.

**Lógica da CF `adminDeleteUser`**

Espelha `accountDeletionService.ts` do client, mas via Admin SDK (bypassa Security Rules):
1. Verifica email do caller (`request.auth.token.email == 'a.thurcos@gmail.com'`)
2. Confirma que o usuário existe no Firebase Auth
3. Coleta refs: workspace pessoal (árvore completa), workspaces de casal criados (+ coupleInvites), membership em workspaces alheios (deleta só o member doc, atualiza o workspace pra remover parceiro), workspaceRefs, billing, privacyRequests, user profile
4. Commita em batches de 450 (limite do Firestore)
5. `auth.deleteUser(uid)` por último — após Firestore limpo
6. Loga `admin_deleted_user` com UID deletado, UID do caller e total de docs

**Permissões Firestore adicionadas**

```
function isAdmin() {
  return signedIn() && request.auth.token.email == 'a.thurcos@gmail.com';
}
```

Read em: `users/{uid}`, `users/{uid}/workspaceRefs/{id}`, `workspaces/{id}` (e toda subárvore via wildcard existente), `coupleInvites/{id}`.

**Confirmação de deleção no UI**

Modal exige digitar o primeiro nome do usuário (não o email, não "EXCLUIR") — mais rápido em uso real, ainda difícil de clicar por acidente. O state local de `users`, `couples` e `invites` é atualizado imediatamente após sucesso; não força reload.

### Setup GCP necessário (one-time)
- Secret Manager API: ativar em console.cloud.google.com (necessário pelo `defineSecret` do billing)
- Cloud Build service account: já tinha a role correta; o erro inicial de build foi race condition entre habilitação de APIs (Eventarc, Cloud Run) e execução da build — resolvido no segundo deploy
- Cleanup policy de Artifact Registry: configurada via `--force`

## 2026-06-17 — Boot resiliente em internet fraca + logos offline

Relato: offline puro funcionava, mas em 5G/rede fraca o app podia ficar parado no "Carregando Zerou..." após refresh; logos SVG de bancos não carregavam offline.

Diagnóstico:
- O Plantão já tinha um padrão de fallback local: libera a UI com cache depois de ~1,8s se Auth/DB não respondem. O Zerou dependia do `onAuthStateChanged` e do snapshot de `users/{uid}` para sair do boot.
- Em rede fraca, diferente de offline puro, o navegador fica em estado "online, mas sem resposta útil". Isso pode atrasar Auth/perfil sem acionar imediatamente o caminho offline.
- O Workbox do Zerou precacheava `js/css/html/ico/png/webmanifest`, mas não `svg`; por isso `public/bank-logos/*.svg` dependia da rede se ainda não estivesse no cache HTTP.

Correções:
- Criado `src/auth/profileCache.ts`: salva o perfil do usuário em `localStorage`, mantém o último UID salvo e tolera cache corrompido.
- `AuthContext` agora usa perfil local imediatamente quando o usuário real do Firebase aparece; depois de 1,8s, se Auth ainda não confirmou mas há perfil local, libera o app com usuário local mínimo (`authFromCache`).
- Falha temporária de snapshot mantém o perfil local; verificação de email e métodos de login ficam bloqueados enquanto a sessão real não é confirmada.
- `vite.config.ts`: `globPatterns` agora inclui `svg`, colocando os logos de banco no precache do PWA.

Validação:
- `npm run typecheck`
- `npm test -- --run` (41/41)

## 2026-06-17 — Correção crítica de funcionamento (travando, offline, zoom)

Relato: app "lindo mas não funciona" — criar conta/meta travava em "pendente" e só sincronizava após refresh; criar meta ficava em spinner infinito; tudo lento; dava para dar zoom e arrastar a tela na horizontal.

Diagnóstico e correções:
- **Transporte do Firestore (causa raiz)**: o WebChannel streaming trava em várias redes/proxies/navegadores, deixando escritas "pendentes" até um reload completo. Ligado `experimentalAutoDetectLongPolling: true` e trocado o cache para `persistentMultipleTabManager` em `src/firebase/config.ts`. Persistência (offline) já existia.
- **UI bloqueando no commit do servidor**: vários fluxos davam `await` na escrita antes de responder. Com o transporte travado, isso virava spinner infinito (ex: criar meta). Refatorado para **escrita otimista** (fire-and-forget + `.catch`): criar/contribuir meta e cofrinho, criar conta. O `onSnapshot` (com persistência) mostra o item na hora com badge pendente e sincroniza depois. Removidos os estados `busy` desses fluxos.
- **Offline-first**: com persistência + auto long-polling + escrita otimista, o app lê do cache offline, enfileira escritas e sincroniza ao voltar — comportamento esperado tipo o plantão.
- **Viewport/zoom/overflow**: `viewport-fit=cover` no `index.html`; `overflow-x: hidden` em html/body/app-main + `text-size-adjust: 100%`; corrigida a margem negativa do `.amount-hero` (era -1.5rem com padding mobile de 1rem → estourava a largura e gerava arraste lateral). Landing verificada sem overflow em 375px.

Convenção que fica: **não bloquear a UI esperando ack do Firestore** — disparar a escrita e deixar o listener refletir (offline-first).

### Offline-first em todo o app (segunda passada)

Generalizado para o app inteiro, não só meta/conta:
- Criado `src/firebase/fireWrite.ts` (dispara a escrita, silencia erro de rede; validação síncrona roda antes e ainda surge ao usuário).
- `financeService`: createCategory/updateCategory/deleteCategory, deleteAccount, deleteGoal, updateBillStatus, softDeleteTransaction, createBill, createRecurringRule agora usam `fireWrite`. createCategory retorna o id na hora (offline seleciona a nova categoria sem esperar).
- `cardService`: createCreditCard, createCardPurchase, recordInvoicePayment, recordInvoiceCredit/Fee (addLedgerOnlyEntry), anticipateInstallments, closeInvoice, reconcileInvoice via `fireWrite` (reads/getDoc continuam com await — leem do cache offline).
- `sharedService`: createSharedExpenseClaim, updateSharedExpenseClaimStatus, createSettlementProposal, acceptSettlement, recordSettlementPayment, addSharedComment via `fireWrite`. Criação de espaço/convite/aceite seguem com await (dependem de rede; não fazem sentido offline).
- Transações (criar/editar) seguem no padrão `Promise.race(350ms)` — já não bloqueiam.
- **Gotcha resolvido**: `orderBy('createdAt')` (serverTimestamp) escondia docs recém-criados offline; removido de goals/goalContributions, ordenação no cliente.
