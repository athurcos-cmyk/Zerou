- `firestore.rules` permite `onboardingGoal` e `onboardingChallenge` opcionais no create do perfil.
- Mensagem genérica removida dos fallbacks de Auth e SharedSpace; onboarding usa `getUserFacingErrorMessage` com texto específico.

## 2026-06-17 — Boot resiliente em internet fraca + logos offline

Relato: offline puro funcionava, mas em 5G/rede fraca o app podia ficar parado no "Carregando Zerou..." após refresh; logos SVG de bancos não carregavam offline.

Diagnóstico:
- O Plantão já tinha um padrão de fallback local: libera a UI com cache depois de ~1,8s se Auth/DB não respondem. O Zerou dependia do `onAuthStateChanged` e do snapshot de `users/{uid}` para sair do boot.
- Em rede fraca, diferente de offline puro, o navegador fica em estado "online, mas sem resposta útil". Isso pode atrasar Auth/perfil sem acionar imediatamente o caminho offline.
- O Workbox do Zerou precacheava `js/css/html/ico/png/webmanifest`, mas não `svg`; por isso `public/bank-logos/*.svg` dependia da rede se ainda não estivesse no cache HTTP.

Correções:
- Criado `src/auth/profileCache.ts`: salva o perfil do usuário em `localStorage`, mantém o último UID salvo e tolera cache corrompido.
- `AuthContext` agora usa perfil local imediatamente quando o usuário real do Firebase aparece; depois de 1,8s, se Auth ainda não confirmou mas há perfil local, libera o app com usuário local mínimo (`authFromCache`).
- Falha temporária de snapshot mantém o perfil local; verificação de email e métodos de login ficam bloqueados enquanto a sessão real não é confirmada.
- `vite.config.ts`: `globPatterns` agora inclui `svg`, colocando os logos de banco no precache do PWA.

Validação:
- `npm run typecheck`
- `npm test -- --run` (41/41)
- `npm run build`
- Conferido `dist/sw.js` com `bank-logos/nubank.svg`, `bank-logos/itau.svg` e `bank-logos/mercado-pago.svg` no precache.

## 2026-06-17 — Correção crítica de funcionamento (travando, offline, zoom)

Relato: app "lindo mas não funciona" — criar conta/meta travava em "pendente" e só sincronizava após refresh; criar meta ficava em spinner infinito; tudo lento; dava para dar zoom e arrastar a tela na horizontal.

Diagnóstico e correções:
- **Transporte do Firestore (causa raiz)**: o WebChannel streaming trava em várias redes/proxies/navegadores, deixando escritas "pendentes" até um reload completo. Ligado `experimentalAutoDetectLongPolling: true` e trocado o cache para `persistentMultipleTabManager` em `src/firebase/config.ts`. Persistência (offline) já existia.
- **UI bloqueando no commit do servidor**: vários fluxos davam `await` na escrita antes de responder. Com o transporte travado, isso virava spinner infinito (ex: criar meta). Refatorado para **escrita otimista** (fire-and-forget + `.catch`): criar/contribuir meta e cofrinho, criar conta. O `onSnapshot` (com persistência) mostra o item na hora com badge pendente e sincroniza depois. Removidos os estados `busy` desses fluxos.
- **Offline-first**: com persistência + auto long-polling + escrita otimista, o app lê do cache offline, enfileira escritas e sincroniza ao voltar — comportamento esperado tipo o plantão.
- **Viewport/zoom/overflow**: `viewport-fit=cover` no `index.html`; `overflow-x: hidden` em html/body/app-main + `text-size-adjust: 100%`; corrigida a margem negativa do `.amount-hero` (era -1.5rem com padding mobile de 1rem → estourava a largura e gerava arraste lateral). Landing verificada sem overflow em 375px.

Convenção que fica: **não bloquear a UI esperando ack do Firestore** — disparar a escrita e deixar o listener refletir (offline-first).

### Offline-first em todo o app (segunda passada)

Generalizado para o app inteiro, não só meta/conta:
- Criado `src/firebase/fireWrite.ts` (dispara a escrita, silencia erro de rede; validação síncrona roda antes e ainda surge ao usuário).
- `financeService`: createCategory/updateCategory/deleteCategory, deleteAccount, deleteGoal, updateBillStatus, softDeleteTransaction, createBill, createRecurringRule agora usam `fireWrite`. createCategory retorna o id na hora (offline seleciona a nova categoria sem esperar).
- `cardService`: createCreditCard, createCardPurchase, recordInvoicePayment, recordInvoiceCredit/Fee (addLedgerOnlyEntry), anticipateInstallments, closeInvoice, reconcileInvoice via `fireWrite` (reads/getDoc continuam com await — leem do cache offline).
- `sharedService`: createSharedExpenseClaim, updateSharedExpenseClaimStatus, createSettlementProposal, acceptSettlement, recordSettlementPayment, addSharedComment via `fireWrite`. Criação de espaço/convite/aceite seguem com await (dependem de rede; não fazem sentido offline).
- Transações (criar/editar) seguem no padrão `Promise.race(350ms)` — já não bloqueiam.
- **Gotcha resolvido**: `orderBy('createdAt')` (serverTimestamp) escondia docs recém-criados offline; removido de goals/goalContributions, ordenação no cliente.
