## 2026-07-14 — Auditoria offline-first: app 100% offline

### Contexto

O app foi construído como offline-first desde o começo (`persistentLocalCache` + `experimentalAutoDetectLongPolling`), mas várias funcionalidades dependiam de internet sem que ninguém percebesse. O dono reportou: "disponível por dia" só aparece com internet, análise de gastos só com internet, "diversas outras coisas".

### Metodologia da auditoria

4 agentes fizeram buscas paralelas por padrões de dependência de rede: `getDocs`/`getDoc` (one-time reads), `onSnapshot` (listeners), `await` em escritas, componentes com estado de loading, e `useEffect` com fetch assíncrono. Cada padrão foi classificado por gravidade. Um agente de review validou as correções depois.

### Problemas encontrados e corrigidos

**Raiz arquitetural #1 — `onSnapshot` nunca dispara com cache vazio + offline**

Firestore com `persistentLocalCache` entrega dados cacheados instantaneamente via `onSnapshot`. Mas se a coleção **nunca foi consultada antes** (cache vazio) e o dispositivo está offline, o listener **nunca dispara** — não há o que entregar. O `subscribeWithBootRetry` (`useFinanceData.ts`) só tratava erros explícitos (`permission-denied`, `unavailable`), não o silêncio do listener.

`useFinanceData` exige que TODAS as 6 coleções disparem antes de `loading = false`. Bastava UMA sem cache pra travar o dashboard inteiro em "Carregando..." pra sempre.

**Correção:** `SLICE_BOOT_TIMEOUT_MS = 2500` — se `onSnapshot` não chamar `onNext` dentro de 2.5s, assume `[]` e destrava. Quando a rede voltar, o listener entrega os dados reais. O `markSliceLoaded` usa `Set.add` (idempotente), então a segunda chamada não corrompe o estado.

**Raiz arquitetural #2 — Escritas com `await` bloqueavam a UI**

5 funções críticas usavam `await setDoc/updateDoc`:
- `createAccount` (criar conta)
- `createTransaction` (lançar despesa/receita)
- `createGoal` (criar meta)
- `contributeToGoal` (contribuir pra meta)
- `updateTransaction` (editar transação)

Todas convertidas para `fireWrite()`. Os IDs são gerados client-side (`createId()`), então retornar imediatamente é seguro. Callers que usavam `Promise.race([write, waitForLocalWrite()])` agora resolvem instantaneamente — o `waitForLocalWrite` (timeout 350ms) era desnecessário porque o Firestore grava no cache local sincronamente.

**3 funções de batch** também convertidas: `coupleGoalDeposit`, `coupleGoalWithdrawal` (batch cross-workspace: cofrinho do casal + transação pessoal), `ensureDefaultCategories`.

**Dashboard "disponível por dia"** (`perDayDisplay`): usava `isCommittedLoading` como guarda e retornava `null` → "Carregando..." mesmo com cache de saldo disponível. Agora usa `cachedSummary.freeToSpendCents` como fallback.

**2 correções adicionais:** `updateAvatarStyle` e `syncAppearanceForUser` convertidas pra `fireWrite`.

**Dead code removido:** `waitForLocalWrite()` + `Promise.race` + `void write.catch()` em `NewTransactionPage` e `EditTransactionPage` (já eram inúteis — `createCardPurchase` sempre usou `fireWrite`). `.catch()` handlers em `CoupleSavingsSection` que nunca disparavam.

### O que NÃO foi alterado (decisão consciente)

### Decisão de design: por que `fireWrite` em vez de mostrar erro?

`CLAUDE.md` estabelece: "fireWrite engole o erro de propósito, por design — não expor erro técnico ao usuário". O `onSnapshot` reflete a mudança imediatamente do cache local (com badge "pendente"). Se o servidor rejeitar (ex.: `permission-denied`), o listener reverte automaticamente. O usuário vê o dado sumir — o que é frustrante, mas menos pior do que um spinner eterno ou uma mensagem técnica incompreensível. A proteção real contra rejeição silenciosa é `npm run test:rules` (regra dos enums/campos novos no mesmo commit).

## 2026-07-14 — Redesign dos avatares + bug real de seleção nunca persistindo

### Por que trocar o estilo

O pedido era só "os avatares estão feios, melhora". Antes de desenhar, comparei visualmente
vários catálogos SVG open-source (baixados via API do DiceBear, renderizados em PNG com
`@resvg/resvg-js` pra inspecionar sem depender de screenshot do browser, que estava instável
nesta sessão): Notionists (ilustração preto/branco, sem tom de pele), Micah e Big Smile (arte
editorial pesada, cores dessaturadas, não combina com o visual flat do resto do app), Adventurer
(overlay de máscara com artefato visual estranho em alguns traços). **Personas** (by Draftbit,
CC BY 4.0) venceu: retrato flat simples, 64×64 viewBox leve, e o schema de opções (`hair`,
`hairColor`, `skinColor`, `eyes`, `mouth`, `nose`, `body`, `clothingColor`) mapeia quase 1:1 pro
que o catálogo antigo já tentava fazer (pele/cabelo/acessório por avatar), só que com arte de
verdade em vez de formas geométricas cruas.

### Geração 100% offline, sem dependência de runtime

`@dicebear/core` + `@dicebear/collection` (9.4.3, pareados — o `core` 10.x mais novo não tem
`collection` compatível ainda) entraram como **devDependencies**, não dependency: a arte é
embutida no pacote, não chama a API do DiceBear nem em build nem em runtime. Mesmo padrão de
`scripts/generate-bank-logos.mjs` (`simple-icons`) — um script gera `public/avatars/*.svg`
estáticos (`npm run generate:avatars`), o app só referencia os arquivos. Cada um dos 12 avatares
tem todos os traços fixados explicitamente no script (não sorteados por seed), pra ser
reproduzível a cada regeração. Cores de roupa reaproveitam a paleta `categoryColors`
(`src/theme/palette.ts`) — mesmo sistema visual das categorias/metas.

`avatarCatalog.tsx` ficou trivial: array de `{id, label, src}` apontando pra
`/avatars/<id>.svg`. Mesmos 12 `id`/rótulo do catálogo antigo (Alegre, Sereno, Radiante...) —
nenhuma migração de dado necessária, já que `avatarStyle` salvo no Firestore é só uma string.
`UserAvatar`/`AvatarPicker` trocaram de `<svg>` inline pra `<img src>`; `avatarSkinTones`,
`avatarHairColors` e as demais constantes de cor do avatar em `palette.ts` foram removidas
(zero literal de cor relacionado a avatar sobrou em `src/`).

### O bug real: `firestore.rules` nunca permitiu `avatarStyle`

Verificação ao vivo no browser (login com `teste@claude.com`) mostrou que clicar num avatar não
tinha efeito nenhum — nem local, nem após reload. Investigação por eliminação: cliques
funcionavam normalmente em outros controles da mesma página (toggle de tema por exemplo),
então não era um problema geral de evento. Chamando o `onClick` da `AvatarPicker` direto via
`props.onClick(...)` e capturando a rejeição da Promise, apareceu o erro real:
`FirebaseError: Missing or insufficient permissions` (`permission-denied`).

Causa: a regra de `allow update` de `users/{uid}` só aceitava três formatos de escrita —
`onlyAppearanceChanged()` (tema), payday, ou `availableMode`. `onlyAppearanceChanged()`'s
`hasOnly([...])` listava `themeMode`, `themeId`, `density`, `fontScale`, `reduceMotion`,
`updatedAt` — **sem `avatarStyle`**. Como `updateAvatarStyle.ts` só escreve
`{avatarStyle, updatedAt}`, o diff do update não batia com nenhum dos três formatos permitidos,
e o servidor rejeitava toda escrita. É o mesmo padrão dos 2 incidentes documentados na REGRA
PRINCIPAL do `CLAUDE.md` (`createdBy` em categoria, `installment_anticipation_credit` no
ledger): campo novo num payload, regra nunca atualizada, e o fire-and-forget do app esconde a
rejeição do servidor completamente — a feature de avatar provavelmente nunca funcionou em
produção desde que foi criada.

Corrigido: `avatarStyle` adicionado ao `hasOnly` de `onlyAppearanceChanged()`, mais uma função
nova `validAvatarStyle()` que valida contra os 12 ids reais do catálogo (ou `null`, pra "sem
avatar/iniciais"). Teste de regressão adicionado em `tests/firestore.rules.test.ts` com o
payload real do cliente (não simplificado, seguindo a lição do incidente do orçamento).
`npm run test:rules` — 49/49. Deploy publicado com autorização explícita do dono
(`npx firebase deploy --only firestore:rules --project zerou-26757`) e confirmado ao vivo:
selecionar um avatar sobrevive a reload completo da página.

De quebra, `AppearanceSettingsPage.handleAvatarChange` estava com `await` bloqueante na
escrita (violava a regra de offline-first do projeto) e engolia qualquer erro sem mostrar nada
ao usuário. Trocado por fire-and-forget (`.catch(...)`) com `FormMessage` exibindo
`getUserFacingErrorMessage` em caso de falha, igual ao padrão usado em `AccountsPage.tsx`.

276 testes unitários, 49 de regra, typecheck e build limpos.

**"Sair deste aparelho" — não era redundante, e isso quase virou uma remoção incorreta**: o
dono pediu pra tirar o botão "porque o Sair já faz isso também". Antes de remover, conferi
`src/auth/authService.ts`: `logout()` simples só chama `signOut(auth)`; `logout({
clearLocalCache: true })` faz isso **e também** `terminate(db)` + `clearIndexedDbPersistence(db)`
+ reload da página — limpa o cache local do Firestore, pensado especificamente pra celular
emprestado/computador compartilhado (não deixar dado financeiro de quem saiu visível pro
próximo). Não são equivalentes. Voltei a pergunta pro dono com a distinção explicada
(`AskUserQuestion`) em vez de remover silenciosamente uma proteção de privacidade real — ele
escolheu fundir num único "Sair" que **sempre** faz a limpeza completa (a opção mais segura das
duas, aplicada a todo logout, não só o caso "aparelho compartilhado").

Implicação que não dava pra ignorar: como o app é offline-first (escreve no cache local antes
de sincronizar), fazer *todo* logout limpar o cache local por padrão significa que sair
enquanto offline com alterações ainda não sincronizadas agora **descarta essas alterações**
silenciosamente — um risco que antes só existia no fluxo opcional "Sair deste aparelho". Pra
mitigar, o texto do diálogo de confirmação (que já existia, mas com copy específico de "aparelho
compartilhado") foi reescrito pra avisar isso: "Se você tiver alterações feitas offline que
ainda não sincronizaram, elas serão perdidas." O menu mobile (`AppShell.tsx`) tinha seu próprio
botão "Sair" separado, usando só `logout()` simples, sem confirmação nenhuma — também trocado
pra usar o mesmo `handleLogout` seguro, por consistência entre desktop e mobile.

De quebra, o nome da pessoa (antes no rodapé, acima do botão Sair) foi movido pro topo da
sidebar, logo abaixo da logo/tagline — pedido explícito do dono na mesma leva de ajustes.

## 2026-07-13 — Grazi: rewrite do contexto, rename Despesas Fixas, nav mobile com avatar

Sessão longa com 3 grandes frentes de trabalho.

### Grazi: rewrite completo do buildFinancialContext

A Grazi original só consultava 4 coleções (categories, transactions expense/card_purchase, bills pending 7 dias, nomes de contas). Isso significava que ela era **cega** para:
- Despesas fixas (coleção `recurring` inteira)
- Faturas de cartão de crédito (`cards/*/invoices`)
- Bills vencidas (`overdue`)
- Saldo real das contas

O Dashboard, em contraste, usa `buildUpcomingCommitments` que agrega bills (pending+overdue) + recurring ativas + invoices com saldo devedor. A Grazi dizia "você não tem nada comprometido" enquanto o Dashboard mostrava R$ 400.

**Rewrite** (`buildFinancialContext.ts`): agora consulta 6 coleções + sub-queries. Output estruturado em 3 seções:

- **RESUMO**: mês atual vs anterior, gasto total + variação %, receitas, saldo em contas (abertura ± transações), total comprometido, livre para gastar
- **GASTOS POR CATEGORIA**: top 5 com comparação mês anterior
- **COMPROMETIDO (próximos 30 dias)**: contas a pagar (pending+overdue, ordenadas VENCIDA primeiro), despesas fixas (recurring ativas com próx. ocorrência), faturas de cartão (open/closed/overdue/partial com saldo devedor). Total quebrado por tipo.

**Prompt atualizado**: regra #3 explícita "Use SEMPRE os dados da seção COMPROMETIDO para responder sobre contas, despesas fixas, faturas e quanto a pessoa deve. Não invente que não tem nada se houver itens listados ali."

**Bugs corrigidos na stessa sessão** (via investigação com 8 agentes em 3 rodadas):
- Índice composto faltando em bills (FAILED_PRECONDITION)
- `card_purchase` invisível (só `expense` era contado)
- BRT timezone ausente (`new Date()` UTC vs `nowInBRT()`)
- Rate limit bypass por concorrência (transação Firestore fail → bypass)
- History sem validação (injeção de role `system`, strings gigantes)
- `??` vs `||` para `competenceMonth`/`categoryId` (string vazia não caía no fallback)
- Null `dueDate` derrubava `buildFinancialContext` inteiro
- Rate limit consumido ANTES do DeepSeek (quota queimada em falha)
- Race condition no incremento do rate limit (`if/else` com `exists` obsoleto)
- Sem validação de API key
- Timeout 15s → 45s + retry único para 429/503

**Testes**: 28 testes em `buildFinancialContext.test.ts` (7 originais + 4 novos: overdue bills, despesas fixas, total comprometido, saldo por conta).

## 2026-07-11 — Conservador pela janela + compra parcelada em andamento (QA ao vivo)

O dono relatou dois pontos numa verificação geral: (1) no conservador, com um cartão de R$5.000 de limite e ~R$3.000 usados, o Disponível ficava negativo, "não faz sentido"; (2) lançar uma compra parcelada que **já estava em andamento** (ex.: óculos 10x, já na parcela 7) era confuso — não havia como.

### 1. Conservador estava contando todas as parcelas de uma vez

Reproduzido por teste com o cenário exato: R$1.000 na conta + R$3.000 em 10x. O conservador (que usava `cutoff = null`, contando **tudo**) dava Comprometido R$3.000 → Disponível **−R$2.000**. Decisão do dono (escolheu a opção "pela data"): **o conservador passa a usar a janela de dias**, nunca assumindo salário. `resolveCommittedCutoff` no modo conservador agora retorna `cutoff = endOfDay(now + committedWindowDays)` e `source = 'window'` (o `source: 'all'` e o `cutoff: null` foram removidos). Assim uma parcela só entra no Comprometido quando o vencimento chega perto — não as 10 juntas. A diferença que restou entre os modos: "até o recebimento" encurta o corte pro dia do salário/receita lançada; "conservador" ignora isso e usa a janela fixa. Pra renda variável (sem payday) os dois coincidem, o que é coerente.

Copy reescrita em três lugares pra refletir isso: `availableModeSummaries` (`availableMode.ts`), o exemplo do mini tutorial (`AvailableModeSheet` — agora "parcela de R$300 que vence em 3 semanas, salário cai antes": conservador guarda a parcela, "até o recebimento" não), e a tela de Recebimento (`PaydaySettingsPage` — a janela de dias deixou de ser "só reserva" e virou o que sempre vale no conservador; a data de recebimento é que fica inativa nesse modo, com recuo visual `.settings-section--inactive`).

## 2026-07-11 — as 3 pendências técnicas, uma de cada vez

### 1. `dashboard` morto no `useFinanceData`

O hook recalculava `calculateDashboardSummary` a cada mudança de transação, **sem faturas, sem payday e sem `availableMode`** — ou seja, um resumo errado — e nenhuma tela o consumia (a `DashboardPage` calcula o seu). Removido. Além do desperdício, era uma armadilha: quem o usasse por engano veria um "Comprometido" que ignora cartão.

### 2. `limit(300)` em `subscribeTransactions`

**A raiz não era o limite, era como se descobre que uma transação foi excluída.** Investigando, apareceram dois problemas distintos.

**Bug maior, offline.** `softDeleteTransaction` grava `deletedAt: serverTimestamp()`, e a regra do Firestore exige `deletedAt == request.time` — então o `serverTimestamp()` é obrigatório. Só que `snapshot.data()`, com as opções padrão, devolve **`null`** para um server timestamp ainda não confirmado. Resultado: offline (ou nos milissegundos antes do ack), `deletedAt` chegava nulo no cache local, a transação continuava no Extrato e a compra continuava somando na fatura. **Num app cujo princípio nº 1 é funcionar offline, a UI desfazia a ação do usuário.** Novo `src/firebase/snapshotData.ts` (`readSnapshotDoc`, `serverTimestamps: 'estimate'`) usado pelos três `withLocalSync` (finance, cards, shared).

**O bug do limite.** `deletedTransactionIds` vinha de `finance.transactions`, a janela das 300 transações mais recentes do workspace. As faturas carregadas cobrem 24 ciclos, então uma compra parcelada de 2 anos atrás continua relevante muito depois de sair da janela — e quando saía, ela sumia do conjunto de excluídas e **o valor voltava a somar na fatura**, que podia até deixar de estar paga.

Solução: `useCardsData` passou a receber `{ knownIds, deletedIds }` e a resolver, com `fetchDeletedTransactionIds` (leitura direta por documento, cacheada), só os `sourceTransactionId` que a janela não cobre — normalmente nenhum. Em erro ou documento ausente, o lançamento **fica**: sumir com ele apagaria uma dívida real. Um teste de mutação confirmou que o teste novo falha quando o fix é desligado.

**Trava de exclusão de conta.** Mesmo mal: `hasTransactions` olhava a janela, então uma conta antiga parecia vazia e podia ser apagada de vez, deixando as transações órfãs no Extrato. Agora `accountHasLiveTransactions` pergunta ao servidor (até 20 documentos por lado, origem e destino) antes de abrir o diálogo, e um erro de rede impede a exclusão em vez de liberá-la. Verificado ao vivo: conta com lançamentos é bloqueada; conta recém-criada abre o diálogo e é apagada.

### 3. Recorrência em dobro

`generateRecurrences` (Cloud Function, 6h) criava a transação com id aleatório e avançava `nextOccurrenceAt`. O botão "Registrar" fazia o mesmo, com outro id aleatório. Quem abrisse o app depois das 6h e clicasse tinha **duas despesas para a mesma ocorrência** — e a recorrência ainda pulava um período.

Duas defesas, em camadas:

- **Id determinístico por ocorrência** (`recurringOccurrenceTransactionId`, espelhado em `functions/src/automation.ts`): `${ruleId}_${data-da-ocorrência}`. As duas escritas caem no mesmo documento; a segunda é um payload de *create* (`version: 1`, `createdAt` novo) numa transação que já existe, e `validTransactionUpdate` a rejeita. O batch inteiro é desfeito — nada é duplicado. Isso virou teste no emulador (`rejects re-creating an existing transaction with the same id`). A data sai em **UTC** (`toISOString`), não no fuso local: o app roda em BRT e a função em UTC, e `getDate()` daria dias diferentes para o mesmo instante — a chave falharia exatamente no caso que existe pra cobrir.
- **A função agora checa se a ocorrência já foi registrada** antes de gravar: se sim, só avança a data e não manda push.
- **A UI parou de oferecer o tiro no pé**: `isRecurrenceDue` esconde o botão "Registrar" quando a próxima ocorrência está no futuro (a linha mostra "Em dia · próxima em 9 ago"). Clicar ali lançava uma despesa que não aconteceu.

Verificado ao vivo: a Netflix vencida mostrou "Registrar", registrou uma única despesa (persistiu após reload, console sem `permission-denied`) e a linha virou "Em dia".

`generateRecurrences` foi deployada (`npx firebase deploy --only functions --project zerou-26757`) com autorização do dono — sem isso, a função antiga continuaria gerando ids aleatórios e só a defesa da UI estaria em pé.

2. **Parcelas colidiam num mês e pulavam outro.** `createCardPurchase` calculava a fatura de cada parcela com `addMonths(purchaseDate, index)`. `addMonths` clampa 31/jan em 28/fev; num cartão que fecha dia 28 (o máximo que a UI permite), o dia clampado deixa de ser "depois do fechamento", então a 2ª parcela caía na MESMA fatura de fevereiro que a 1ª — e março ficava sem parcela nenhuma. Uma compra 4x em 31/01/2026 produzia `['2026-02', '2026-02', '2026-04', '2026-05']`. Novo `resolveInstallmentCycle` conta os meses a partir do mês da primeira fatura, ancorado no dia 1 (nunca somando meses à data da compra). `resolveInvoiceCycle` virou o caso `index = 0`.

3. **Antecipação oferecia faturas passadas.** O filtro de "parcelas futuras" em `InvoicePage` era `inv.id !== invoiceId` — qualquer fatura ainda não paga entrava, inclusive uma vencida. Antecipá-la creditaria a fatura velha e debitaria a atual: jogar dívida pra frente, o oposto de antecipar. Agora o critério é `referenceMonth > referenceMonth da fatura atual`.

4. **Antecipação escondia parcelas irmãs.** Todas as parcelas de uma compra parcelada compartilham o mesmo `sourceTransactionId` (existe uma transação `card_purchase` só). O código guardava os já-antecipados num `Set` desse id, então antecipar uma parcela sumia com todas as irmãs da mesma fatura. Trocado por contagem de ocorrências. A lógica saiu de dentro do componente para `src/cards/anticipation.ts` (`selectAnticipatableInstallments`), com 10 testes — era exatamente o tipo de regra que ficou meses quebrada sem ninguém notar.

5. **Antecipar numa fatura fechada/paga.** O painel aparecia em qualquer fatura. O débito da antecipação entraria num ciclo já encerrado. Agora só na fatura aberta.

6. **Compromisso que vence no próprio dia do salário sumia do Comprometido.** `nextPaydayFrom` devolve meia-noite; contas e faturas são gravadas ao meio-dia (`fromDateInputValue`). Comparando instantes, uma conta que vencia no dia do pagamento ficava depois do corte. Pior: `addDays(now, 30)` herda a hora atual, então o Comprometido mudava conforme a hora em que o app era aberto. O corte agora é sempre `endOfDay`. `findNextIncomeDate` também passou a ignorar receita datada de hoje (já está no saldo) em vez de compará-la com o instante `now`.

7. **Push "Fatura fechada: R$ 0,00".** `invoices/{id}.outstandingBalanceCents` nasce `0` em `invoicePayload` e **nunca é atualizado** — todos os totais são derivados do ledger no cliente (`useCardsData`). A Cloud Function `closeInvoicesDue` lia o campo cru, então toda notificação de fatura fechada anunciava R$ 0,00. Agora calcula o saldo a partir do ledger (`invoiceOutstandingCents`, espelha `calculateInvoice`). **Precisa de `firebase deploy --only functions`.**

## 2026-07-08 — Texto preto ilegível nos temas escuros: Tailwind v4 não roda o preflight legado

Pedido do dono: "ao mudar de temas muitas letras ficam com a cor preta". Reproduzi em Obsidian e fui checando página por página até achar um caso concreto: em `/app/accounts`, o `<h2>Adicionar conta financeira</h2>` (dentro do botão que expande o formulário) tinha `color: rgb(0, 0, 0)` — preto sólido — apesar de `--text-primary` do tema resolver pra `#F3F4F6` (quase branco).

### Investigação

Descartei hipóteses na ordem:
1. **Token de cor errado por tema?** Não — inspecionei `themes.css` pros 4 temas escuros, `--text-primary` está correto em todos.
2. **Cor hardcoded em algum componente?** Não — `noHardcodedColors.test.ts` já garante que não existe hex/rgba literal fora de `themes.css`/`palette.ts`, e passava.
3. **Regra CSS explícita pintando de preto?** Não — busquei via CSSOM (`document.styleSheets`, recursivo por `@layer`/`@media`) por qualquer regra com `color` que desse `match()` no elemento ou nos ancestrais. Nenhuma.
4. **Cadeia de herança quebrada?** Aqui sim: inspecionei o `color` computado subindo a árvore (`h2` → `div` → `button` → `form` → ...). O `<form>` tinha a cor certa (clara); o `<button>`, filho direto, já vinha preto — e o `<button>` não tem nenhuma classe, só `style` inline sem `color`.

Isso só faz sentido se o **preflight do Tailwind não estivesse rodando** (é o preflight que normalmente aplica `button, input, select, textarea { color: inherit; ... }`, porque browsers não herdam `color` em controles de formulário por padrão — usam a cor de sistema `buttontext`, que resolve pra preto). Busquei o texto `color: inherit` em todas as ~992 regras CSS carregadas (incluindo dentro de `@layer`): zero ocorrências.

Causa confirmada: `global.css` abre com `@tailwind base; @tailwind components; @tailwind utilities;` — sintaxe **v3**. O projeto está em Tailwind v4 (`"tailwindcss": "^4.3.1"`, `@tailwindcss/postcss`), que espera `@import "tailwindcss";` e não processa mais as diretivas `@tailwind` legadas. O preflight inteiro (incluindo o reset de `color` em controles de formulário) nunca era injetado — só não dava mais visível porque quase todo o app usa classes próprias com `color` explícito; os únicos pontos expostos eram elementos nativos sem classe, como esse `<h2>` dentro de `<button>` sem estilo.

**Escopo real**: o mesmo padrão (`<button>` sem classe envolvendo um `<h2 style={{margin:0}}>` de "abrir formulário colapsável") existe em 5 páginas — `AccountsPage`, `CardsPage`, `BillsPage`, `GoalsPage`, `SharedSpacePage`. Explica o "muitas letras" do pedido original: não era um componente isolado, era um gap estrutural que atingia qualquer heading/texto dentro de um `<button>`/`<input>`/`<select>`/`<textarea>` sem classe própria.

### Fix

Adicionado um reset explícito no topo de `global.css`, logo após o `* { box-sizing: border-box }`:

```css
button,
input,
select,
textarea {
  font: inherit;
  color: inherit;
}
```

Decisão: **não** migrei `@tailwind base/components/utilities` para `@import "tailwindcss"` nesta sessão. Migrar o entrypoint do Tailwind é uma mudança de pipeline de build com superfície maior (afeta como o v4 faz content-scanning e gera utilitários) e o app não usa praticamente nenhuma classe utilitária Tailwind — é 100% CSS próprio orientado a tokens. Resolver o sintoma real com uma regra CSS padrão, sem tocar no restante do preflight (que também não roda, mas nada mais na UI dependia dele visivelmente), foi a correção mais cirúrgica. Migrar o import do Tailwind fica como possível item futuro se surgir mais alguma lacuna do preflight.

### Validação

Testado ao vivo nos 4 temas escuros (troquei de tema, revisitei Contas/Cartões com o formulário aberto) e confirmei nos 2 temas claros que nada regrediu (já estavam corretos por coincidência, já que preto é a cor certa lá). `npm run typecheck` e os 129 testes (incluindo `noHardcodedColors`) seguem passando — o fix não introduz nenhum literal de cor, só herança.

## 2026-07-07 — Auditoria de uso do Firestore

Motivada por uma pergunta direta do dono ao ver o painel de métricas do Firestore ("o que usamos muito que não seja necessário?"), antes de tirar o app do modo "só eu uso" e ligar de vez pra mais gente.

### Como investiguei

Levantei todo ponto de leitura/escrita do app: todo `onSnapshot` (listeners ao vivo), todo `getDocs`/`getDoc` (leituras avulsas), toda `collectionGroup` (usada só nas Cloud Functions agendadas), e todo write que roda em mount/boot em vez de responder a uma ação explícita do usuário. Fiz isso lendo o código, não adivinhando — cada achado abaixo eu confirmei rastreando quem chama o quê antes de reportar.

### Achado principal: fatura + ledger sem limite, crescimento composto

`useCardsData.ts` monta um listener de faturas por cartão (`subscribeInvoices`) e, pra cada fatura carregada, um listener de ledger próprio (`subscribeInvoiceLedger`). Nenhuma das duas queries tinha `.limit()`. Pior: como os efeitos dependem de `cardIds`/`invoiceIds` (string com todos os ids concatenados), toda fatura nova desmonta e remonta **todos** os listeners de ledger já existentes — não é incremental.

Um usuário com 3 cartões ativos há 3 anos facilmente chegaria a 100+ listeners simultâneos, a maioria de faturas já pagas (`buildUpcomingCommitments` só usa faturas `closed`/`open` recentes pro cálculo de "Comprometido" — faturas antigas e quitadas não entram em nenhuma conta).

**Fix aplicado**: `subscribeInvoices` agora tem `limit(24)` (~2 anos de ciclo mensal por cartão). Cogitei uma solução mais sofisticada (listener ao vivo só pra fatura `open`/`closed`, leitura avulsa pra fatura `paid`/`overpaid` já resolvida), que preservaria histórico infinito navegável sem custo crescente — mas é bem mais código tocando um fluxo financeiro sensível, pra um produto que ainda não tem usuário real. Prefiro a versão simples agora; se um dia o histórico de +2 anos precisar ser navegável de novo, aí sim vale o esforço maior.

### Achado zero-risco: feature de comentários nunca finalizada

`subscribeSharedComments` estava sempre montado (listener ao vivo) toda sessão com espaço compartilhado ativo, pra uma coleção `comments` dentro do workspace do casal. Rastreei ponta a ponta: existia até uma função `addSharedComment` (writer) — mas **nenhuma tela em lugar nenhum a chama**, e **nenhuma tela lê `shared.comments`**. Parece ter sido um scaffold de feature que nunca foi ligado a nenhum botão.

Removido de ponta a ponta, não só o listener:
- `SharedComment` (tipo, `contracts.ts`)
- `addSharedCommentSchema` / `AddSharedCommentInput` (`sharedSchemas.ts`)
- `commentsRef`, `addSharedComment`, `subscribeSharedComments` (`sharedService.ts`)
- Campo `comments` do estado + subscription (`useSharedWorkspaceData.ts`)
- `validSharedCommentCreate` + `match /comments/{commentId}` (`firestore.rules`)

### Achado de baixo impacto: token FCM regravado toda sessão

`requestAndRegisterPushToken()` fazia `setDoc` do token toda vez que o app abria (uma vez por sessão), mesmo quando o token e a plataforma não tinham mudado desde a última vez — o FCM token é estável entre sessões na prática, só muda em reinstalação/limpeza de dados. Criado `src/pwa/pushTokenCache.ts` (mesmo padrão de `profileCache.ts`, cache local por uid) — só grava no Firestore quando o token realmente muda.

### Convites expirados — TTL nativo, não Cloud Function

Discussão à parte (mesma sessão): convites do casal (`coupleInvites`) que nunca são aceitos nem revogados ficavam parados no banco pra sempre além do prazo de 48h — só existia limpeza manual (botão "Limpar expirados", escondido, nunca usado na prática) e limpeza automática só nos casos de aceitar/revogar/cancelar espaço/excluir conta. Em vez de escrever mais uma Cloud Function agendada, configuramos uma **política de TTL nativa do Firestore** no campo `expiresAt` da coleção `coupleInvites` — zero código, zero manutenção, o Firestore apaga sozinho em até ~24-72h após a expiração, inclusive retroativo aos documentos que já existiam. Feito manualmente no Google Cloud Console (não há `gcloud` nesta máquina nem comando equivalente no `firebase-tools` pra automatizar isso).

### Guia de referência: quando adicionar `.limit()` numa coleção

O dono perguntou especificamente sobre como pensar em "quando escalonar" pras coleções que ficaram sem limite mas de baixo risco (`sharedExpenseClaims`, `settlements`, `bills`, `goals`, `recurring`, `accounts`, `categories`). A resposta documentada em `SESSAO.md`:

- O custo real de um `onSnapshot` é o tamanho da **leitura inicial** (1 leitura por doc que a query traz na montagem do listener) — atualizações depois disso só custam 1 leitura por doc que de fato mudou, não uma releitura de tudo.
- Regra de bolso: quando uma coleção de um único workspace passar de **~500-1000 documentos**, cada abertura de tela custa centenas de leituras — é aí que compensa `.limit()` + paginação. Abaixo disso, é complexidade sem ganho.
- Por taxa de crescimento real: `accounts`/`categories`/`goals`/`recurring` nunca chegam lá (limitados pelo comportamento humano — ninguém cria centenas). `bills` cresce devagar (uma recorrência por mês). `sharedExpenseClaims`/`settlements` do casal são as que mais podem crescer com uso contínuo (um casal ativo pode gerar 20-50 despesas divididas/mês — 1.000-2.000 em poucos anos).
- Sinal de alerta pra monitorar no painel do Firestore: **leitura por usuário ativo** subindo ao longo do tempo — não leitura total (que sobe naturalmente com a base de usuários). Se o "por usuário" subir, alguma coleção está envelhecendo mal.

### Decisão de design: `type` no lugar de valor com sinal

O padrão já existente pra metas **pessoais** (`contributeToGoalWithTransaction`) usa `amountCents` com sinal (delta positivo/negativo direto no `increment()`). 1. Trocar a validação pra `validSignedMoneyCents` (já existe na regra, usada hoje só em `openingBalanceCents`) e aceitar `amountCents` negativo.

Fui de (2) — é o padrão já estabelecido no domínio (`InvoiceLedgerEntryType` usa `type: 'purchase' | 'payment' | 'refund_credit' | ...` pra decidir o sinal em vez de valores assinados; ver `calculateInvoice.ts`). ## 2026-07-09 — `adminDeleteUser` sem permissão de invocação pública (erro "internal" no admin)

O dono reportou: na tela de admin, digitar `EXCLUIR` e confirmar a exclusão de uma conta sempre falhava, mostrando "internal" em vermelho.

### Diagnóstico

Primeira hipótese (descartada): comparação estrita sem `.trim()` no `DeleteConfirmModal` (`AdminPage.tsx`) deixando o botão travado por um espaço acidental no input. Corrigido por precaução (mesmo padrão de `LoginMethodsPage.tsx:76`), mas o dono confirmou que digitava certo e via "internal" — ou seja, a chamada *estava* sendo disparada.

"internal" no SDK do Firebase é o sintoma de uma resposta que o cliente não conseguiu interpretar como erro de função (não é o formato JSON `{error: {status, message}}` que `HttpsError` produz). Isso apontou pra fora do código da function, não pra dentro.

Comparado os dois endpoints callable de admin sem nenhuma autenticação (`curl` direto):

- `adminForceLogout` → `{"error":{"message":"Acesso negado.","status":"PERMISSION_DENIED"}}` — JSON de aplicação. A requisição chega no código, roda `assertAdmin()`, nega corretamente.
- `adminDeleteUser` → HTML bruto do Google Frontend: `"Your client does not have permission to get URL /adminDeleteUser from this server."` — bloqueio de infraestrutura, a requisição nunca chega no código.

Confirmado nos logs de produção (`npx firebase functions:log --project zerou-26757`): duas tentativas reais em 2026-07-09 03:13 UTC (bem próximas — provavelmente os dois cliques do dono) rejeitadas com `"The request was not authenticated. Either allow unauthenticated invocations or set the proper Authorization header. Empty Authorization header value."`.

Causa raiz confirmada consultando a IAM policy do Cloud Run direto pela API (`run.googleapis.com/v2/.../services/admindeleteuser:getIamPolicy`, usando o access token já em cache do `firebase-tools` local — sem precisar de `gcloud`):

- `admindeleteuser` → policy **vazia**, nenhum binding.
- `adminforcelogout` → `{"role": "roles/run.invoker", "members": ["allUsers"]}`.

Callable functions do Firebase fazem a própria autenticação *dentro* do código (via token do Firebase Auth) — pra isso funcionar, o serviço Cloud Run por baixo precisa aceitar invocação pública (`allUsers` como `run.invoker`); é assim que `adminForceLogout` sempre funcionou. `adminDeleteUser` perdeu esse binding, provavelmente no redeploy que resolveu o conflito de dois codebases reivindicando o mesmo nome (2026-07-07, ver entrada acima) — mover uma function entre codebases recria o recurso, e a recriação nem sempre reaplica o binding de IAM automaticamente.

### Fix

Um `npx firebase deploy --only functions:admin:adminDeleteUser` sozinho **não resolveu** — deploy de update em function já existente não reaplica IAM policy (só deploy de criação faz isso, aparentemente). O fix real foi chamar a API do Cloud Run direto:

```
POST https://run.googleapis.com/v2/projects/zerou-26757/locations/southamerica-east1/services/admindeleteuser:setIamPolicy
{"policy":{"bindings":[{"role":"roles/run.invoker","members":["allUsers"]}]}}
```

Autenticado com o access token já salvo em `~/.config/configstore/firebase-tools.json` (token do login do `firebase-tools`, escopo suficiente sem precisar instalar `gcloud`). Verificado depois: `adminDeleteUser` sem auth agora retorna o mesmo JSON `PERMISSION_DENIED` de aplicação que `adminForceLogout` — sinal de que a requisição chega no código de novo.

### Lição pra não repetir

Se um dia `adminDeleteUser` (ou qualquer outra callable) for recriada — movida de codebase, deletada e redeployada, renomeada — **checar a IAM policy do Cloud Run depois**, não assumir que o redeploy cuidou disso sozinho. Sintoma característico: erro genérico `"internal"` no cliente em vez do erro de aplicação esperado (`permission-denied`, etc.), e no `curl` direto sem auth, resposta HTML do Google Frontend em vez de JSON.

### Validação

`npm run typecheck` limpo. Testado via `curl` direto nos dois endpoints (raw Cloud Run URL e `cloudfunctions.net`) antes/depois do fix — resposta mudou de 403 HTML pra JSON de aplicação nos dois. Não foi possível testar o fluxo completo de exclusão via UI (exigiria deletar uma conta real de produção); o dono deve confirmar que o botão funciona na próxima tentativa.

## 2026-07-09 — Revisão de design da Dashboard (`/ui-ux-pro-max`) + 3 fixes

O dono pediu uma análise da tela inicial em foco mobile via skill `/ui-ux-pro-max`: o design tem identidade própria ou é "genérico de IA"? Análise feita lendo `DashboardPage.tsx`, `themes.css`, `global.css` e `docs/design/DESIGN.md`, comparando com o baseline "genérico" que o próprio banco de dados do `/ui-ux-pro-max` recomendaria pra um fintech dashboard sem contexto (slate escuro `#0F172A`, roxo/azul "trust", Inter/IBM Plex Sans). Veredito: autoral, não template — cor quente incomum (tangerina sobre areia), par tipográfico específico (DM Sans 800 + Instrument Sans), gradiente/sombra de marca com intenção hierárquica, ilustrações SVG próprias nos vazios, tile colorido por categoria (não um accent único repetido).

Login feito com a conta de teste fornecida pelo dono (`teste@claude.com`) direto no `localhost`, pra tirar screenshot real em vez de só analisar código — confirmou a análise e revelou um bug visual que só aparecia renderizado.

### Bug encontrado: ícone de categoria descentralizado

Nos screenshots reais, o desenho dentro do quadrado colorido de categoria (`.category-mark`) aparecia encostado no canto superior-esquerdo em vez de centralizado — mas o quadrado e a cor de fundo estavam certos.

Diagnóstico via `getComputedStyle` no preview: `.category-mark` tinha `display: block` computado, não o `display: grid` declarado no CSS. Causa: `.category-mark` é um `<span>` dentro de `.list-row`, e a regra `.list-row strong, .list-row span { display: block; }` (`global.css`, existe pra empilhar título/subtítulo em duas linhas nas outras linhas de lista) tem especificidade maior (`0,1,1`, classe + elemento) que `.category-mark` sozinho (`0,1,0`, só classe) — vencia mesmo vindo antes no arquivo. Sem essa regra concorrente sendo óbvia (é de uso geral, não pensada pra ícones), o bug não aparece em nenhuma leitura estática do componente isolado.

Fix: seletor virou `span.category-mark` — mesma especificidade (`0,1,1`) da regra concorrente, e como já vem depois no arquivo, o empate é resolvido a favor dele. Não precisou tocar na regra genérica.

### 3 melhorias sugeridas e implementadas na Dashboard

Depois da análise, o dono pediu opinião de melhorias. Três sugeridas, todas implementadas:

1. **Datas cruas**: `toDateInputValue` (`financeDates.ts`, formato `yyyy-MM-dd` — feito pra `<input type="date">`) estava sendo reusado como texto de tela em "Últimos movimentos" e "Próximos compromissos", mostrando "2026-07-08" pro usuário. Novo helper `formatFriendlyDate` — "Hoje", "Ontem", ou `d MMM`/`d MMM yyyy` com `locale: ptBR` do date-fns (pedido explícito do dono: nomes de mês em português, não inglês — sem o `locale` o `format` do date-fns cai no inglês por padrão). Aplicado nos dois pontos da Dashboard.
2. **Vazio de "Próximos compromissos"** era só um `<p>` seco enquanto "Transações recentes" ao lado usava `EmptyState` ilustrado — inconsistência entre dois cards lado a lado. Nova ilustração `'bills'` em `EmptyState.tsx` (calendário com tarja laranja + círculo de check), seguindo a mesma gramática visual das outras 5 (retângulos/círculos com os tokens do tema, sem texto embutido no SVG).
3. **"Resumo de gastos"** não tinha o tile de cor por categoria que "Últimos movimentos" tem, perdendo o "fio visual" entre os dois cards. Precisou trocar o agrupamento de gasto por categoria de **nome** pra **ID** (`spendingByCategory` antes agrupava por `category.name`, string; agora por `category.id`) pra conseguir resolver o objeto categoria completo e renderizar `<CategoryMark category={category} />` ao lado do nome. `categoryNames` (map auxiliar só de nomes) ficou sem uso e foi removido.

### Pendência sinalizada, não implementada

O mesmo mau uso de `toDateInputValue` como texto de exibição existe em `TransactionsPage.tsx`, `BillsPage.tsx`, `InvoicePage.tsx`, `CardDetailPage.tsx`, `CardsPage.tsx`, `RecurringPage.tsx` e `SearchPage.tsx` — fora do escopo pedido (só Dashboard). Sinalizado como chip de sugestão (`spawn_task`) em vez de expandir o escopo sem pedido.

### Validação

`npm run typecheck` limpo, `npm test -- src` (128/128). Confirmado visualmente no preview com a conta de teste: `display: grid` computado de volta no `.category-mark`, datas mostrando "Ontem"/"8 jul", ilustração nova renderizando, ícone de categoria aparecendo no resumo de gastos.

## 2026-07-09 — Criar categoria nova falhava silenciosamente + auditoria de regras do Firestore

O dono reportou dois sintomas ao lançar uma transação e criar categoria pelo picker: (1) criar a categoria também salvava a transação incompleta, e (2) a categoria criada não existia mais ao voltar pra tela.

### Bug 1 — form aninhado num portal propaga o submit

`CategoryField.tsx` (usado dentro de `<form>` em 4 telas: `NewTransactionPage`, `EditTransactionPage`, `BillsPage`, `RecurringPage`) renderiza seu form de "Nova categoria" dentro de um `BottomSheet`, que usa `createPortal`. Fisicamente o form fica em outro lugar do DOM, mas **continua filho do form externo na árvore React** — e eventos sintéticos do React fazem bubbling pela árvore React, não pela árvore DOM (comportamento documentado do React sobre portais). Sem `event.stopPropagation()`, `event.preventDefault()` sozinho não impedia o submit do form da categoria de também disparar o `onSubmit` do form pai, salvando a transação/conta/recorrência incompleta. Fix: `event.stopPropagation()` adicionado ao `handleSubmit` do form de categoria.

### Bug 2 — regra do Firestore desatualizada há ~3 semanas

Com o bug 1 corrigido, o segundo sintoma persistiu: criar "Pets" voltava pra lista como se tivesse funcionado, mas recarregando a página a categoria sumia. Comparando `createCategory` (`financeService.ts`) com `validCategoryCreate` (`firestore.rules`): o campo `createdBy` foi adicionado no payload do cliente no commit `9a62edf` (17/06), mas a regra (escrita em `93456e5`, 14/06) nunca foi atualizada — `keys().hasOnly([...])` rejeitava qualquer doc com uma chave fora da lista, incluindo `createdBy`. Toda criação de categoria personalizada era rejeitada pelo servidor, e o padrão fire-and-forget do app (`fireWrite`) suprime esse erro de propósito — ninguém via nada quebrar.

**Achado colateral**: o helper de teste `categoryPayload` (`tests/firestore.rules.test.ts`) também não incluía `createdBy` — por isso o teste de regras sempre passou mesmo com o app quebrado em produção. Corrigido junto (agora recebe `uid` e inclui `createdBy`).

Fix aplicado em `validCategoryCreate`: `createdBy` adicionado ao `hasOnly()` e validado (`== request.auth.uid`). Deploy: `npx firebase deploy --only firestore:rules --project zerou-26757`.

### Bug 3 — regressão própria: o fix do bug 2 quebrou o seeding de categorias padrão

Antes de considerar o trabalho concluído, percebi que `request.resource.data.createdBy == request.auth.uid` sem condição também se aplica à criação das categorias **padrão** (`ensureDefaultCategories` → `buildDefaultCategory`, `financeService.ts`/`defaultCategories.ts`) — que nunca envia `createdBy` (categorias padrão não têm dono, só `isDefault: true`). Esse caminho ficaria quebrado pra qualquer workspace novo a partir do primeiro deploy, sem ter sido notado ainda (o workspace de teste já tinha os defaults seedados de antes). Corrigido antes de reportar como concluído: a regra agora exige `isDefault == true && sem createdBy` OU `isDefault == false && createdBy == uid` — mutuamente exclusivo, reflete exatamente os dois caminhos reais do cliente. Redeployado.

### Auditoria completa: todo write do app vs. toda regra

Depois dos 3 bugs, o dono pediu uma busca por "falhas como essa e falhas que possam vir a ter". Comparei sistematicamente cada função de escrita (`setDoc`/`updateDoc`/`batch.set`/`batch.update`) em `financeService.ts`, `cardService.ts`, `sharedService.ts`, `workspaceService.ts`, `appearanceSync.ts`, `notifications.ts` e `accountDeletionService.ts` contra o `hasOnly()`/`diff().affectedKeys().hasOnly()` correspondente em `firestore.rules` (~35 funções de validação).

Nenhum outro desalinhamento encontrado — todas as ~30 outras funções de escrita batem exatamente com suas regras: `createAccount`, `createTransaction`/`updateTransaction`, `createGoal`/contribuições de meta, `createBill`, `createRecurringRule`, `createCreditCard`/compra parcelada/pagamento de fatura (ledger), criação/aceite/revogação de convite do casal, claims e settlements, fundação de usuário/workspace no onboarding (`ensurePersonalFoundation`), sync de tema (`onlyAppearanceChanged`), tokens de push (`fcmTokens`, sem validação de campo — `allow read, write: if isSelf(uid)`).

**Achado colateral que não é bug**: os campos de totais da fatura (`purchasesTotalCents`, `outstandingBalanceCents` etc.) nunca são escritos de volta no Firestore pelo cliente — `validInvoiceUpdate` só permite mudar `status`/`updatedAt`. Não é uma lacuna: esses totais são recalculados no cliente a partir do ledger a cada leitura (`calculateInvoice`, `useCardsData.ts`), design intencional, não meia-implementação esquecida.

**Teste de regressão adicionado**: `tests/firestore.rules.test.ts` ganhou um teste cobrindo os dois ramos da regra de categoria (seed sem `createdBy` sucede se `isDefault: true`; falha se `isDefault: false` sem `createdBy`; falha se `isDefault: true` com `createdBy`).

### Validação

`npm run typecheck`, `npm test -- src` (128/128), `npm run build` — todos limpos. Não foi possível rodar `npm run test:rules` (Java não instalado localmente — já era problema conhecido, ver `SESSAO.md`/`TODOS.md`). Testado ao vivo em produção: criei duas categorias novas ("Pets", depois "TesteFinal") direto no picker da tela de nova transação, ambas persistiram após reload da página, com a regra final já deployada.

## 2026-07-09 — QA completa de cartões/faturas/Comprometido: 4 bugs reais encontrados e corrigidos + feature de payday

Sessão longa, disparada por um relato concreto do dono: lançou uma despesa no cartão, foi para o Extrato e excluiu — a despesa sumiu da lista, mas continuou aparecendo na fatura. A partir daí, pediu para testar totalmente a lógica do programa (login com conta de teste), depois foi aprofundando em "Comprometido"/"Disponível" até pedir uma revisão final de cartões/faturas/recorrências.

### Bug 1 — exclusão de compra no cartão não saía da fatura

`createCardPurchase` grava a transação (`transactions/{id}`) e o(s) lançamento(s) de ledger da fatura (`cards/{cardId}/invoices/{invoiceId}/ledger/{entryId}`) como registros separados. O botão de excluir em Transações (`softDeleteTransaction`) só marca `deletedAt` na transação — nunca toca no ledger, e as regras do Firestore não deixam um membro comum apagar/editar ledger entries (só na exclusão total da conta). Resultado: excluir a compra sumia do Extrato, mas o valor ficava preso na fatura pra sempre (pior em compra parcelada — ledger espalhado em várias faturas futuras).

Fix sem mudar regra: `useCardsData.ts` passou a receber o conjunto de IDs de transações excluídas e filtra os ledger entries correspondentes (via `sourceTransactionId`) antes de calcular saldo/lista da fatura.

Achados na mesma investigação:
- **"Fatura atual" errada**: `CardDetailPage`/`CardsPage` pegavam a primeira fatura `open` na ordem de chegada do array (desc por `referenceMonth`) em vez da mais próxima de fechar — com parcelamento, mostrava uma fatura futura de R$0 no lugar da que estava realmente acumulando a compra nova, e "Pagar agora" pagaria a fatura errada. Novo helper `pickCurrentInvoice` (`src/cards/cardDates.ts`).
- Editar uma transação `card_purchase` pelo formulário genérico corrompia o registro (perdia `cardId`/`type`). Botão "Editar" escondido pra esse tipo; acesso direto pela URL mostra aviso.
- Confirmação (`ConfirmDialog`) adicionada antes de excluir qualquer transação — não existia nenhuma.
- Nomes diferentes mobile x desktop pra mesma rota ("Extrato"/"Casal" no mobile vs "Transações"/"Compartilhado" no desktop) — unificado em `AppShell.tsx`.
- Bug visual: "Saldo total" quebrava linha entre o "-" e o valor quando negativo (UAX#14 permite quebra depois de hífen seguido de letra) — `white-space: nowrap` + teto do `clamp()` reduzido em `global.css`.

### Bug 2 — fatura de cartão contava no Comprometido um mês antes da hora

Testando o cenário "fatura de R$60 que só vence mês que vem, mas tenho R$1.400 na conta — não quer dizer que R$60 está comprometido", achei dois problemas encadeados:

1. `resolveInvoiceCycle` (`cardDates.ts`) não tratava cartão com `dueDay < closingDay` — padrão comum no Brasil (fecha dia 25, vence dia 5 do mês seguinte). Uma compra feita antes do fechamento calculava vencimento no MESMO mês do fechamento, podendo cair antes até da própria compra. Corrigido: quando `dueDay < closingDay`, o vencimento rola pro mês seguinte ao ciclo.
2. Mesmo com a data certa, `buildUpcomingCommitments` (`financeCalculations.ts`) decidia "isso é Comprometido" por `referenceMonth <= mês atual` (mês do CICLO da compra) em vez da data de vencimento real — contando a fatura inteira um mês antes da hora pra cartões desse tipo. Trocado, por decisão do dono, pra usar o mesmo cutoff de contas a pagar/recorrências (antes do próximo salário registrado, ou N dias); fatura `closed` continua contando sempre.

Testado ao vivo: fatura que vence **antes** do salário lançado continua contando; fatura que vence **depois** sai do Comprometido na hora (caiu de R$2.039,90 pra R$639,90 no teste real).

### Feature — "quando você recebe?" (payday), em duas rodadas

Sem receita futura lançada na mão, o cutoff acima caía num chute genérico fixo. Implementei uma pergunta no onboarding (4º passo, opcional) — dia fixo do mês / Xº dia útil / fim do mês — guardada em `users/{uid}.payday` e usada como 2º fallback do cutoff (`nextPaydayFrom`, `src/finance/payday.ts`), com tela `Configurações → Recebimento` pra editar depois.

O dono pediu revisão: a pergunta não servia pra quem tem renda variável (o próprio caso dele — plantonista, sem salário fixo todo mês). Adicionei uma 4ª opção explícita "Renda variável" (`PaydayRule` ganhou `{type: 'variable_income'}`, gravado como escolha real, não ausência de resposta) e uma **janela de dias configurável** (`committedWindowDays`, padrão 30, editável por qualquer pessoa em Configurações) substituindo o `30` fixo escondido no código. O Dashboard passou a explicar o número: legenda embaixo de "Comprometido" mostrando se está considerando receita lançada, payday fixo, ou a janela de dias.

Bug pego no meio da implementação: a tela de Configurações resincronizava do perfil a cada render, fazendo o rádio "descasar" no clique antes da escrita otimista voltar do Firestore — mesmo padrão de `hasLocalOverride` de `appearance.store.ts` resolveu.

Regras do Firestore (`validPayday`, `validCommittedWindowDays`, `onlyPaydaySettingsChanged`) escritas e revisadas manualmente (Java local quebrado bloqueia `npm run test:rules`), deployadas com autorização explícita do dono, e verificadas ao vivo em produção depois de cada deploy.

### Revisão final — o bug mais sério da sessão

Pedido do dono: revisar todo o comportamento de Comprometido/Disponível/Recorrente/Faturas/Cartão mais uma vez. Achados:

1. **Antecipação de parcelas nunca funcionou em produção**: `validInvoiceLedgerEntryType` (`firestore.rules`) nunca incluiu `'installment_anticipation_credit'` na lista de tipos aceitos — só o débito estava lá. O tipo já existia no TypeScript (`InvoiceLedgerEntryType`) desde a criação da feature, mas a regra nunca foi atualizada — mesmo padrão do bug de `createdBy`/categoria. Toda tentativa de "Confirmar antecipação" era rejeitada silenciosamente pelo servidor desde sempre. Corrigido e implantado; confirmado ao vivo (fatura atual +R$300, fatura futura zerada, limite total inalterado).
2. **Débito de antecipação fantasma**: mesmo corrigindo a regra, o débito de antecipação era gravado como um único lançamento somado sem `sourceTransactionId` — excluir a compra original depois de antecipada deixava esse débito preso na fatura pra sempre (o crédito da fatura futura era limpo corretamente, o débito não). Corrigido: um lançamento de débito por parcela antecipada, cada um com o `sourceTransactionId` da compra correspondente (`cardService.ts`). Confirmado ao vivo: antecipar + excluir a compra original não deixa mais nenhum valor preso.
3. **Recorrências**: mensagem "A próxima ocorrência avança para [data atual] + 1 período" no BottomSheet de pagamento mostrava a data ATUAL sem avançar, com um sufixo que não calculava nada — agora usa `nextOccurrenceDate` de verdade (`RecurringPage.tsx`).
4. **Varredura sistemática**: todos os campos tipo-enum do app comparados contra as regras do Firestore (tipo de transação, status de fatura, tipo de conta, status de bill, frequência de recorrência, bandeira de cartão, tipo de meta) — nenhum outro desalinhamento encontrado.
5. **Observação não corrigida** (feature faltando, não bug): `Bill.status` tem `'overdue'` no schema, mas nada no app jamais marca uma conta a pagar como vencida — fica "Pendente" pra sempre. O cálculo já conta certo, só falta indicação visual.

### Validação final

`npm run typecheck` e `npm test` (147/147) limpos em cada etapa. `npm run test:rules` seguiu bloqueado o dia inteiro pelo Java quebrado localmente — toda mudança de regra (payday, variable_income, committedWindowDays, installment_anticipation_credit) foi revisada só por leitura manual de sintaxe + teste unitário do payload, deployada com autorização explícita do dono, e verificada ao vivo em produção logo depois de cada deploy. Essa pendência (`CLAUDE.md`) já causou dois incidentes reais na mesma sessão — prioridade alta pra resolver antes da próxima mudança de regra.
