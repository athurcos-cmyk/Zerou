### 1.2 Lacunas encontradas no dominio de Dependencias

**LACUNA-1: Funcionamento do `npm audit` em subprojetos nao foi demonstrado.**
O relatorio afirma "npm audit root rodado -- 13 vulns" e "npm audit functions rodado -- 8 vulns", mas nao apresenta o comando executado nem o raw output. Para functions-admin, o numero e aproximado ("~8 (transitivas do firebase-admin)") sem confirmacao exata. O revisor deveria ter incluido os comandos `cd functions && npm audit` e `cd functions-admin && npm audit` no relatorio ou pelo menos os JSONs de auditoria.

**LACUNA-2: Tabela resumo contradiz analise do DEP-5.**
A tabela final marca DEP-5 como "Nao (nao explotavel)" na coluna "Afeta Producao?" mas o texto do proprio DEP-5 diz "Sim (functions)" -- as functions sao producao. Ha uma contradicao material: o revisor confunde "nao explotavel" (juizo de risco) com "nao afeta producao" (juizo de alcance). Sao dimensoes diferentes.

**LACUNA-3: Auditoria de `functions-admin/package.json` nao verificou scripts lifecycle.**
O DEP-12 cobre root e menciona node_modules geral, mas nao verifica explicitamente se `functions-admin/package.json` tem lifecycle scripts. Os subprojetos tem menos dependencias, entao o risco e menor, mas a auditoria deveria ter verificado cada um individualmente.

**LACUNA-4: Nao verificou `node_modules/.package-lock.json`.**
A partir do npm v8+, ha um `node_modules/.package-lock.json` interno que o npm usa para install. A auditoria nao verificou se este arquivo e consistente com o `package-lock.json` top-level. Discrepancias entre os dois podem indicar instalacao corrompida ou ataque de substituicao de pacote.

### 2.2 Lacunas encontradas no dominio React

**LACUNA-1: Nao auditou o padrao de cleanup nos 12+ arquivos que usam `onSnapshot`.**
O revisor afirma que o cleanup e "sistematico" mas nao verifica cada um dos 12 arquivos que usam `onSnapshot`. Por exemplo:
- `src/auth/AuthContext.tsx` -- sera que o `onSnapshot` do perfil e limpo no unmount?
- `src/settings/LoginMethodsPage.tsx` -- tem cleanup?
- `src/settings/accountDeletion.store.ts` -- cleanup adequado em store global?

Uma verificacao amostral mostrou que `useFinanceData.ts` tem cleanup exemplar (`cancelled`, `timers`, `unsubscribers`), mas outros arquivos mereceriam escrutinio.

**LACUNA-2: Nao auditou estados de carregamento concorrentes em `Promise.all`.**
O `AdminPage.loadAll()` usa `Promise.all` para 4 operacoes. Se uma falha e outra resolve, o `setState` no `.finally` mistura dados parciais com erro. Isso nao foi discutido.

**LACUNA-3: Nao verificou se `fireWrite` esta sendo usado corretamente em TODOS os services.**
O relatorio cita `fireWrite` como padrao, mas nao verifica se ha operacoes de escrita no Firestore que NAO usam `fireWrite` (i.e., `await setDoc(...)` ou `await updateDoc(...)` sem o wrapper). Essas seriam violacoes do padrao offline-first.

**LACUNA-4: Nao auditou a funcao `renderAssistantMessage` para vazamento de informacao.**
Alem do XSS (REACT-02), `renderAssistantMessage` faz renderizacao de markdown que pode expor dados financeiros do prompt se a Cloud Function retornar conteudo malformatado. O revisor focou no vetor de ataque (XSS) mas nao considerou o vazamento de dados via feedback loop (usuario ve dados financeiros no chat, que sao renderizados).

**LACUNA-5: Nao testou se o StrictMode do React 19 quebra algo.**
O `main.tsx` envolve o app em `<StrictMode>`. React 19 StrictMode monta/desmonta componentes em desenvolvimento para detectar efeitos sem cleanup. Se algum `useEffect` depende de mount para enviar analytics ou inicializar estado, isso pode disparar duplicado em dev -- algo que o revisor nao menciona.

### Lacunas entre dominios (achados que um dominio deveria ter passado pro outro)

1. **Seguranca -> React**: O XSS da Assistente (REACT-02) e uma vulnerabilidade de seguranca, nao apenas de arquitetura React. O relatorio de Dependencias (08) deveria ter mencionado a falta de sanitizacao de output como um risco de suply chain indireto (a Cloud Function e controlada pelo projeto, mas o codigo que renderiza o output e do client).

2. **Design System -> UX**: A falta de animacao de transicao de paginas (SOL-09) e um problema de UX que deveria estar no relatorio 16-ux.md. Da mesma forma, o arredondamento excessivo (UX-10) e um problema de design system que deveria estar em 15-sol.md.

3. **React -> UX**: A ausencia de Error Boundary (REACT-01) tem impacto UX direto: o usuario ve tela branca em vez de uma mensagem amigavel. O relatorio UX deveria ter capturado isso como um estado de erro nao tratado.

### Recomendacoes para a Camada 3 (se houver)

1. Validar se `npm audit` foi executado corretamente nos 3 subprojetos (rodar novamente)
2. Verificar `renderAssistantMessage` com DOMPurify ou substituir por parse sem innerHTML
3. Verificar os estados de erro em todas as telas (o que o usuario ve quando `useFinanceData().error` nao e null)
4. Rodar `npm test -- --run src/test/noHardcodedColors.test.ts` e confirmar output
5. Auditoria visual dos 6 temas nao verificados (Floresta, Lavanda, Rosa, Noturno, Carbono, Ametista)
6. Testar a11y basico: Tab order, focus ring, contraste em tema claro e escuro
