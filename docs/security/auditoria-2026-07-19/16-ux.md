## Hierarquia visual

### UX-01 — Saldo total perde protagonismo no mobile (Dashboard)
**Severidade:** Média
**Local:** `DashboardPage.tsx`, `.dash-summary` (global.css:692-697)
**Descrição:** No desktop, o Saldo Total ocupa a esquerda do `dash-summary` com `1.2fr` contra `2fr` dos secundários — proporção razoável. No mobile (breakpoint 900px), `dash-summary` vira uma coluna única (global.css:3079-3081), e a ordem é: Saldo Total, Disponível, Comprometido. O Saldo Total (hero) perde o contraste de escala: seu `clamp(2rem, 4.8vw, 2.35rem)` fica muito próximo do `clamp(1.3rem, 2.8vw, 1.75rem)` do Disponível/Comprometido. O gradiente brand ainda destaca, mas a diferença de tamanho é pequena.
**Impacto:** O "número herói" (dinheiro) não se impõe visualmente no mobile como deveria — os três cards parecem ter pesos equivalentes, quando o Saldo Total deveria ser claramente o principal.
**Solução sugerida:** Aumentar o `font-size` do `.dash-balance .display-number` no mobile para algo como `clamp(2.2rem, 7vw, 2.35rem)`.
**Confiança:** 8

### UX-02 — "Seu resumo" como título da página principal é genérico
**Severidade:** Baixa
**Local:** `DashboardPage.tsx:320`
**Descrição:** O `h1` da página principal é "Seu resumo". Não comunica valor — é um placeholder administrativo. O `eyebrow` "Olá, {nome}" é mais personalizado, mas o título principal não diz o que o usuário está vendo.
**Impacto:** Perde a oportunidade de reforçar a proposta de valor na tela mais importante.
**Solução sugerida:** Substituir por algo como "Seu dinheiro hoje" ou "Visão do mês".
**Confiança:** 6

### UX-03 — CTAs do dashboard têm pesos visuais desiguais
**Severidade:** Baixa
**Local:** `DashboardPage.tsx:372-388`, `.quick-actions` (global.css:889-898)
**Descrição:** O grid `quick-actions` coloca "Lançar agora" (`.button--primary`) na primeira linha inteira, e os quatro atalhos (Contas, Cartões, Compromissos, Metas) como `.button--subtle` abaixo. Isso é consistente no desktop. No mobile, o grid inteiro some e o `dash-shortcut-row` (2x2) aparece sem o "Lançar agora" (redundante com o FAB). O problema: "Lançar agora" é o CTA principal do app, mas no mobile ele SOME da tela. Se o usuário não sabe que o FAB central é "Lançar", ele pode não encontrar a ação primária.
**Impacto:** Ação primária do app não tem label fixa visível no mobile; o FAB é só um ícone `+`.
**Solução sugerida:** Considerar um tooltip ou micro-label no FAB. O DESIGN.md já define que o FAB é "tangerina para Lançar", mas o usuário novo não tem esse contexto.
**Confiança:** 7

## Consistência entre telas

### UX-04 — Contas a Receber não tem seção de "recorrentes" como Contas a Pagar
**Severidade:** Média
**Local:** `BillsPage.tsx` vs `ReceivablesPage.tsx`
**Descrição:** BillsPage tem duas seções: "Recorrentes" (assinaturas/contas fixas) + "Compromissos" (avulsas). ReceivablesPage tem apenas uma lista plana com filtros (A receber/Recebidos/Todos). Não há suporte a recebimentos recorrentes. Se o usuário tem um freela mensal, precisa recriar manualmente todo mês.
**Impacto:** Funcionalidade incompleta para receitas recorrentes — quebra a paridade de "espelho" declarada no changelog.
**Solução sugerida:** Adicionar seção de recorrentes em Contas a Receber, similar à de BillsPage.
**Confiança:** 9

### UX-05 — Headers de página inconsistentes
**Severidade:** Média
**Local:** Múltiplas páginas
**Descrição:**
- Dashboard: `h1` "Seu resumo", `eyebrow` "Olá, {nome}"
- Contas: `h1` "Contas", `eyebrow` "Pessoal"
- Cartões: `h1` "Cartões", `eyebrow` "Pessoal"
- Transações: `h1` "Transações", `eyebrow` "Pessoal"
- Contas a Pagar: `h1` "Contas a Pagar", `eyebrow` "O que você tem pra pagar"
- Contas a Receber: `h1` "Contas a Receber", `eyebrow` "O que você tem pra receber"

O padrão varia: páginas financeiras (Contas, Cartões, Transações) usam `eyebrow` "Pessoal", que é genérico e não agrega valor. Contas a Pagar/Receber usam eyebrow descritivo ("O que você tem pra pagar/receber"), que é mais útil.
**Impacto:** Inconsistência na voz — algumas páginas tem eyebrow descritivo, outras tem "Pessoal" que não diz nada.
**Solução sugerida:** Unificar o padrão para eyebrow descritivo em todas as páginas.
**Confiança:** 8

### UX-06 — Página de Patrimônio Líquido está sem entrada de navegação
**Severidade:** Informativa
**Local:** `NetWorthPage.tsx`, `AppShell.tsx:92` (comentário)
**Descrição:** A página de Patrimônio Líquido existe, está funcional, mas não tem link na sidebar nem no menu mobile (desativado em 2026-07-16 por pedido do dono). O usuário só acessa se souber a URL. A página usa `EmptyState` com ilustração `wallet`, o título da página usa `page-title--compact` sem a classe base `page-title` (linha 122: `className="page-title--compact"` — sem `page-title`), o que faz o `h1` perder o estilo base (margin reset, `font-size` do `clamp`).
**Impacto:** Página órfã e subtilmente quebrada (heading sem classe base). Usuário não encontra.
**Solução sugerida:** Adicionar `page-title` na classe. Decidir se a página volta à navegação ou é removida.
**Confiança:** 10

## Estados de UI

### UX-07 — Empty state de faturas usa markup genérico em vez de EmptyState
**Severidade:** Baixa
**Local:** `CardDetailPage.tsx:251-257`
**Descrição:** O empty state do histórico de faturas não usa o componente `EmptyState` — usa um `div.empty-copy` com `span.empty-icon` e um SVG cru do `CreditCard`. Todas as outras listas vazias do app usam `EmptyState` com ilustrações SVG próprias. Este é o único que destoa.
**Código:**
```tsx
<div className="empty-copy">
  <span className="empty-icon">
    <CreditCard size={24} aria-hidden="true" />
  </span>
  <p className="text-secondary">Nenhuma fatura ainda...</p>
</div>
```
**Impacto:** Inconsistência visual perceptível. Um card que tem um empty state ilustrado ao lado (vazio de cartões) contrasta com o texto seco de faturas vazias.
**Solução sugerida:** Substituir por `<EmptyState illustration="cards" compact ... />`.
**Confiança:** 9

### UX-08 — Loading state do NetWorthPage é texto seco
**Severidade:** Média
**Local:** `NetWorthPage.tsx:99-107`
**Descrição:** Enquanto carrega, a página mostra um `<div>` com padding e o texto "Carregando..." centralizado. Não é um skeleton, não é um spinner — é texto puro. Todas as outras telas lidam com loading de forma mais elegante (dashboard usa cache view).
**Impacto:** Experiência pobre em conexão lenta — o usuário vê "Carregando..." em fonte básica.
**Solução sugerida:** Usar um skeleton simples ou pelo menos o mesmo padrão de fallback do dashboard.
**Confiança:** 8

### UX-09 — SyncStatusBadge pode causar ruído visual em listas longas
**Severidade:** Baixa
**Local:** `BillsPage.tsx`, `ReceivablesPage.tsx`, `TransactionsPage.tsx`
**Descrição:** Cada linha de item (conta, transação, receivable) mostra um `SyncStatusBadge`. Em uma lista com 20+ itens onde a maioria está "synced", são 20+ badges verdes repetidos que competem com a informação principal (descrição + valor). É informação de diagnóstico, não de uso cotidiano.
**Impacto:** Ruído visual em listas longas — o olho é puxado para múltiplos pontos verdes em vez do valor e descrição.
**Solução sugerida:** Mostrar o badge apenas em itens com status `pending` ou `error`. Itens synced podem ficar silenciosos.
**Confiança:** 6

## "AI slop" visual

### UX-10 — Arredondamento excessivo em elementos de formulário
**Severidade:** Baixa
**Local:** `.input` / `.select` (global.css:402, `border-radius: 1rem`)
**Descrição:** Inputs têm `border-radius: 1rem` (16px). Selects também. Chips têm `border-radius: 999px` (pílula). Botões têm `border-radius: 12px`. Embora não seja pílula em tudo, o `1rem` nos inputs é alto demais para campos de texto — parece mais um campo de busca do que um input de formulário financeiro.
**Impacto:** Visual levemente "genérico de SaaS" — inputs muito arredondados passam a impressão de ferramenta casual, não de app financeiro sério.
**Solução sugerida:** Reduzir `border-radius` dos inputs para 10px (0.65rem).
**Confiança:** 5

### UX-11 — Sombra do FAB é propositalmente dramática
**Severidade:** Informativa
**Local:** `--shadow-fab: 0 8px 20px rgba(238, 85, 36, 0.4)` (themes.css:40)
**Descrição:** O FAB tem uma sombra laranja forte `0 8px 20px rgba(238,85,36,0.4)`, que é intencional para destacar o CTA principal. Não é "AI slop" porque tem propósito claro (destacar a ação de lançar). Mantido como informativa — é um caso de sombra justificada pelo contexto.
**Impacto:** Nenhum (é apropriado).
**Confiança:** 10

## Mobile-first

### UX-12 — Filtros de Transações: chips "Tudo/Despesas/Receitas/Transferências" rolam mas não há indicador de scroll
**Severidade:** Baixa
**Local:** `TransactionsPage.tsx:226-244`, `.chip-row--scroll` (global.css:3294-3302)
**Descrição:** Os chips de tipo (Tudo, Despesas, Receitas, Transferências) usam `.chip-row--scroll` com `overflow-x: auto` e `scrollbar-width: none`. Em 375px, "Transferências" pode ficar parcialmente cortado, e não há sombra/indicador de que o conteúdo continua para a direita. O design usa o "corte seco" como affordance (comentário no CSS), mas um gradiente sutil na borda direita melhoraria a descoberta.
**Impacto:** Usuário pode não perceber que há mais opções de filtro.
**Solução sugerida:** Adicionar `mask-image` linear-gradient nas bordas para dar dica visual de scroll.
**Confiança:** 5

### UX-13 — Touch targets em listas: apenas "Contas a Pagar" tem problemas potenciais
**Severidade:** Baixa
**Local:** `BillsPage.tsx` — seção de recorrentes
**Descrição:** Na lista de recorrentes, cada linha tem 3-4 botões inline: "Registrar"/"Pagar adiantado", Editar (lápis), Desativar (X). Em mobile (375px), esses alvos ficam muito próximos em `list-row-end` com `gap: 0.35rem`. O `icon-button` tem 16px de ícone sem padding explícito visível no CSS — o touch target pode ser menor que 44px recomendado.
**Impacto:** Erros de toque em listas de contas a pagar/recorrentes.
**Solução sugerida:** Garantir que `.icon-button` tenha no mínimo 44x44px de touch target.
**Confiança:** 7

### UX-14 — BottomSheet ocupa 88vh com scroll interno
**Severidade:** Informativa
**Local:** `.sheet-panel` (global.css:3596-3608, `max-height: 88vh`)
**Descrição:** Sheets têm altura máxima de 88vh com `overflow-y: auto` no `.sheet-body`. O swipe-to-dismiss é funcional e bem implementado (threshold + flick detection, restrito ao grabber/header). A altura de 88vh deixa espaço para ver o background, o que é um bom padrão mobile. Mantido como informativa.
**Impacto:** Nenhum — implementação adequada.
**Confiança:** 9

### UX-15 — Formulário de nova transação: hero gigante e teclado mobile
**Severidade:** Alta
**Local:** `NewTransactionPage.tsx`, `.amount-hero` (global.css:3141)
**Descrição:** O header de valor usa `font-size: clamp(2.6rem, 11vw, 3.4rem)` — um valor grande e imponente. Contudo, quando o teclado numérico abre no mobile, o hero pode ocupar quase metade da viewport (o valor + tipo + botão voltar). O form abaixo tem fields que podem ficar encobertos pelo teclado. O `entry-actions` é sticky com offset `--bottom-nav-space`, o que ajuda, mas o CategoryField (que abre um BottomSheet) e SelectFields podem ficar atrás do teclado antes de abrir a sheet.
**Impacto:** Em telas muito pequenas (iPhone SE, 375px), o campo "Título" pode ficar logo acima do teclado, e os seletores (conta, categoria) podem exigir scroll.
**Solução sugerida:** Verificar se `CategoryField` e `SelectField` (que abrem sheets) são acessíveis com teclado aberto. O botão "Salvar" sticky ajuda, mas campos acima dele podem sumir.
**Confiança:** 6

## Desktop

### UX-16 — Sidebar não tem scroll de conteúdo próprio
**Severidade:** Baixa
**Local:** `AppShell.tsx`, `.sidebar` (global.css:566-578)
**Descrição:** A sidebar é um container flex com `height: 100vh` e `overflow-y: auto`. Com o BrandLockup, 11 links, título "Conta", 5 links de configuração + botão "Como funciona" + "Sair", o conteúdo pode não caber na viewport em notebooks (768px de altura). O scroll vertical resolve, mas os links de navegação principais (que são mais importantes) podem sumir abaixo da dobra.
**Impacto:** Em telas menores, links importantes como "Compartilhado" ou "Assistente" podem ficar abaixo da dobra enquanto "Sair" está sempre no final.
**Solução sugerida:** Considerar agrupar "Conta" e configurações em um sub-menu ou accordion.
**Confiança:** 6

### UX-17 — Dashboard desktop desperdiça espaço horizontal
**Severidade:** Baixa
**Local:** `DashboardPage.tsx`, `page-content` (global.css:616-619: `width: min(100%, 72rem)`)
**Descrição:** O conteúdo do dashboard é limitado a 72rem (~1152px). Em monitores wide (1920px+), sobram ~768px de espaço vazio nas laterais. As listas de compromissos e transações recentes ficam em uma coluna estreita dentro do `finance-grid` (2 colunas no desktop, mas é só um número pequeno de compromissos).
**Impacto:** Subaproveitamento de espaço horizontal — poderia mostrar mais informação.

## Micro-interações

### UX-21 — Transição entre telas de lançamento não tem animação
**Severidade:** Baixa
**Local:** `NewTransactionPage.tsx` (navegação via `navigate()`)
**Descrição:** Ao criar uma transação ou compra no cartão, o usuário é redirecionado via `navigate()` sem animação de transição. O hero gigante desaparece e a tela de lista aparece instantaneamente. Não há micro-interação de sucesso (além do dado aparecer na lista).
**Impacto:** Falta de feedback de sucesso pós-ação — a transição é seca.
**Solução sugerida:** Adicionar um toast/notificação "Transação salva" ou animar a saída do hero.
**Confiança:** 6

### UX-22 — Swipe-to-dismiss no BottomSheet implementado, mas sem indicação visual de que é arrastável
**Severidade:** Baixa
**Local:** `BottomSheet.tsx`, `sheet-drag-zone`/`sheet-grabber`
**Descrição:** O grabber (barrinha horizontal no topo) existe e o swipe-to-dismiss funciona. Porém, não há feedback visual enquanto o usuário arrasta (mudança de opacidade no backdrop, cor do grabber). Apenas a transição `translateY` no painel.
**Impacto:** O usuário pode não perceber que pode arrastar para fechar.
**Solução sugerida:** Escurecer o backdrop proporcionalmente ao arrasto, ou mudar a cor do grabber.
**Confiança:** 5

## Copy/UX writing

### UX-23 — "Conclua seu cadastro inicial" aparece em formulários
**Severidade:** Média
**Local:** `BillsPage.tsx:193`, `ReceivablesPage.tsx:63`, `AccountsPage.tsx:63`, `NewTransactionPage.tsx:97`
**Descrição:** Quando `workspaceId` ou `user` não estão prontos, a mensagem de erro é "Conclua seu cadastro inicial antes de criar contas/registrar transações". Isso aparece se o Firestore ainda não terminou de carregar o perfil — é um estado de loading, não de cadastro incompleto. O usuário pode pensar que seu cadastro está incompleto quando na verdade é só latência.
**Impacto:** Mensagem enganosa — pode confundir o usuário.
**Solução sugerida:** Algo como "Aguarde um momento..." ou tratar esse estado antes de habilitar o formulário.
**Confiança:** 8

### UX-24 — "Pessoal" como eyebrow é genérico
**Severidade:** Baixa
**Local:** `TransactionsPage.tsx:203`, `CardsPage.tsx:68`, `AccountsPage.tsx:135`

**Impacto:** Perde oportunidade de microcopy útil.
**Solução sugerida:** Substituir por algo descritivo específico de cada página.
**Confiança:** 7

### UX-25 — "Compromissos" como atalho no dashboard confunde
**Severidade:** Média
**Local:** `DashboardPage.tsx`, links "Compromissos" para `/app/bills`
**Descrição:** Tanto o `quick-actions` quanto o `dash-shortcut-row` têm um link "Compromissos" que leva para Contas a Pagar. O termo "Compromissos" não existe na navegação principal — a sidebar tem "Contas a Pagar" e "Contas a Receber". Um usuário que clica em "Compromissos" espera ver ambos (contas a pagar + a receber), mas vê apenas o que tem a pagar.
**Impacto:** Quebra de expectativa — o label não corresponde ao destino.
**Solução sugerida:** Renomear para "A Pagar" ou "Contas" para alinhar com a navegação principal.
**Confiança:** 7

## Comparação Contas a Pagar vs Contas a Receber

### UX-28 — BillsPage tem sugestões de serviço (ServicePicker), ReceivablesPage não
**Severidade:** Baixa
**Local:** `BillsPage.tsx:278-290` vs `ReceivablesPage.tsx`
**Descrição:** Ao criar uma conta a pagar, o usuário vê sugestões de serviços (energia, aluguel, internet) com ícones. Ao criar um a receber, não há sugestões equivalentes — apenas campos de texto. A sugestão de serviço ajuda a preencher descrição e categoria de uma vez, o que é útil.
**Impacto:** Disparidade de experiência — ReceivablesPage tem menos "ajuda" no preenchimento.
**Solução sugerida:** Adicionar sugestões de "quem paga" (cliente, freela, etc.) similar ao ServicePicker.
**Confiança:** 5

### UX-29 — Filtros diferentes entre Contas a Pagar e Contas a Receber
**Severidade:** Baixa
**Local:** `BillsPage.tsx:38-45` vs `ReceivablesPage.tsx:20-26`
**Descrição:** Contas a Pagar tem 4 filtros: Em aberto, Vencidas, Pagas, Todas. Contas a Receber tem 3: A receber, Recebidos, Todos. A diferença faz sentido semântico (não há "vencidas" em recebimentos porque o overdue é tratado junto com pending), mas a label "A receber" vs "Em aberto" é sutilmente diferente.
**Impacto:** Mínimo — apenas nota de inconsistência de nomenclatura.
**Solução sugerida:** Considerar "A vencer" / "Em aberto" para alinhar as labels.
**Confiança:** 4

### UX-30 — BillsPage usa ServiceMark nas listas; ReceivablesPage usa CategoryMark com fallback fixo
**Severidade:** Baixa
**Local:** `BillsPage.tsx:447` (ServiceMark) vs `ReceivablesPage.tsx:219` (CategoryMark)
**Descrição:** Na lista de contas a pagar, cada item mostra `ServiceMark` (ícone do serviço/assinatura). Em recebimentos, usa `CategoryMark` com fallback fixo `{ icon: 'money', color: defaultCategoryColors.income_salary }`. O tratamento visual é diferente: pagáveis têm ícones variados (Netflix, Steam, etc.), recebíveis têm sempre o mesmo ícone de dinheiro. A lista de recebíveis fica visualmente monótona.
**Impacto:** Lista de recebíveis é menos rica visualmente que a de pagáveis.
**Solução sugerida:** Extrair `fromWho` para um nome de contato/instituição e usar uma marca similar a ServiceMark para recebíveis.
**Confiança:** 6

## Tema escuro (Dark modes)

### UX-31 — Não foi possível auditar temas escuros sem acesso visual
**Severidade:** Informativa
**Local:** `themes.css` (Obsidian, Midnight etc. — não lidos na íntegra)
**Descrição:** O sistema tem 6 temas via `data-theme`. A auditoria cobriu apenas Paper (Sol) e parcialmente Perola. **Impacto:** Não auditar dark mode é uma lacuna. As imagens/ilustrações dos empty states provavelmente funcionam (usam variáveis CSS, não cores fixas). Mas não é possível confirmar contraste em dark sem ver as telas.
**Solução sugerida:** Auditoria visual complementar com tema Obsidian ativo.
**Confiança:** 3

### Padrões observados

- **Pontos fortes:** Sistema de design coeso (tokens, temas, componentes), implementação cuidadosa de offline-first, swipe-to-dismiss funcional, empty states ilustrados na maioria das telas, feedback de erro amigável com `getUserFacingErrorMessage`, boa acessibilidade (`aria-*`, `role`, `inputMode`).
- **Fragilidades:** Inconsistência entre páginas "espelho" (Bills vs Receivables), alguns estados de UI negligenciados (loading de NetWorthPage, empty state de faturas), mensagens de erro que podem confundir, labels de navegação que não correspondem aos destinos.
- **Riscos visuais:** Baixo — não há "AI slop" relevante. As sombras e arredondamentos são propositais e controlados pelo design system.
